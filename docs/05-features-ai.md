# AI Features

## Overview

The extension integrates multiple AI providers as a **tool-using assistant** layered on top of connected MCP servers.

The AI system is responsible for:
- natural language chat
- dynamic tool selection
- multi-step workflows across one or more MCP servers
- clear visibility into what tools were called and why

The AI system is **not** responsible for knowing Jira or Confluence APIs directly.
Those capabilities arrive through MCP tool discovery.

---

## Supported AI Providers

Users bring their own API keys.

| Provider | API Style | Default Role in Product |
|----------|-----------|-------------------------|
| OpenAI | chat / tool calling / streaming | general assistant |
| Gemini | content generation + tool calling | general assistant |
| Claude | messages + tool use | general assistant |

The extension keeps provider support inside the background service worker and talks to provider APIs directly.

---

## Unified AI Adapter Interface

```typescript
interface AIAdapter {
  readonly provider: "gemini" | "openai" | "claude";
  chat(params: ChatParams): Promise<ChatResponse>;
  chatStream(params: ChatParams): AsyncIterable<ChatStreamChunk>;
}

interface ChatParams {
  messages: ChatMessage[];
  tools?: ToolDefinition[];
  systemPrompt?: string;
  temperature?: number;
  maxTokens?: number;
}
```

The AI layer does not care whether a tool came from Atlassian, GitHub, Filesystem, or another MCP server.
It only consumes the normalized tool list generated by the Tool Broker.

---

## Dynamic Tool Discovery

### Source of Truth

For each connected server, the extension discovers tools by:

1. opening or creating an MCP connection
2. running initialization
3. calling `tools/list`

The returned tools are normalized into one internal catalog.

### Namespaced Tool IDs

To avoid collisions, each discovered tool is exposed to the model with a namespaced identifier.

Example:
- upstream server id: `atlassian-main`
- upstream tool: `jira_search`
- AI-visible tool id: `atlassian-main__jira_search`

This lets the extension connect multiple servers that expose overlapping tool names.

### Tool Visibility

The tool list shown to AI is filtered by:
- connected server status
- server enabled/disabled flag
- per-tool allowlist / denylist
- preset defaults
- current chat mode, if the UI narrows the scope

---

## Tool Broker Contract

The AI service never calls MCP servers directly.

It talks to the Tool Broker, which is responsible for:
- mapping namespaced tool ids back to `{ serverId, originalToolName }`
- validating confirmation policy
- invoking `callTool`
- capturing tool results and errors in a consistent format

Example internal lookup:

```typescript
type BrokerExecutionRequest = {
  namespacedToolName: string;
  input: Record<string, unknown>;
  source: "ai" | "quick-action" | "preset-flow";
};
```

## Chat Loop

```
1. User sends message
2. AI Service builds request:
   - system prompt
   - recent chat history
   - connected server summaries
   - normalized tool definitions
3. Provider starts streaming text and/or tool calls
4. If a tool call appears:
   - Tool Broker validates it
   - confirmation is requested if needed
   - MCP call executes through the correct server
   - result is fed back to the model
5. Final answer is streamed to the UI
```

The number of tool iterations per turn should remain capped to avoid loops and runaway costs.

## System Prompt Design

The system prompt should describe:
- which servers are connected
- which ones are healthy
- which tool families are available
- any policy restrictions
- whether the session is read-only

Example sketch:

```text
You are an assistant inside a Chrome extension.

Connected servers:
- atlassian-main (preset: Atlassian, status: connected)
- github-team (custom MCP, status: connected)

Rules:
1. Only use tools that are currently available.
2. Prefer read operations before write operations.
3. Ask for confirmation before destructive changes.
4. If a required tool is unavailable, say so clearly.
```

This is intentionally server-aware, not product-API-aware.

## Safety Controls

### Confirmation Policy

The broker may require confirmation before executing:
- delete-like tools
- bulk mutation tools
- tools explicitly marked high-risk by preset rules

### Read-Only Mode

A server may be connected in a read-only profile.
When that happens:
- write tools are omitted from AI
- preset UIs downgrade to browse/search only

### Server Isolation

Each tool call is attributed to one server.
The UI should show:
- server name
- tool name
- tool input summary
- success / error state

This matters when the user connects more than one MCP server.

---

## Streaming UX

The side panel should show:
- assistant text streaming
- tool call cards as they happen
- waiting / running / done states
- failure details when a tool call fails

This is especially important when one prompt spans multiple server calls.

---

## Preset-Aware Guidance

Although the AI layer is generic, presets can add better hints.

For the Atlassian preset, the extension may add prompt hints like:
- prefer search before issue mutation
- show issue keys prominently
- treat page deletion as destructive

These are lightweight hints, not hard-coded protocol branches.

## Example Workflows

### Atlassian Only

User asks:
- "Find my open P1 bugs and summarize the top three"

Likely tool flow:
1. `atlassian-main__jira_search`
2. optional `atlassian-main__jira_get_issue`

### Cross-Server

User asks:
- "Summarize this sprint and draft release notes"

Possible flow:
1. Jira tools from Atlassian server to gather issue data
2. Confluence tools from the same server to draft a page
3. In future, another documentation or Git server could join the same workflow

The AI layer should not care whether these tools come from one server or many.

## Future Extensions

### MCP Resources

Later phases may let the assistant read server-provided resources for:
- richer grounding
- cached reference material
- preset navigation experiences

### MCP Prompts

Later phases may consume server-provided prompts when helpful, especially for:
- guided setup
- preset workflows
- domain-specific templates

These should be additive, not blockers for the core MVP.
